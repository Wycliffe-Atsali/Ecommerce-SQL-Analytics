# PostgreSQL Implementation Notes

## Overview

This document summarizes the implementation process used to build the PostgreSQL database for the Brazilian E-Commerce Public Dataset by Olist.

Rather than simply importing CSV files, the objective was to implement a normalized relational database while validating the schema, preserving data integrity, and documenting implementation decisions. Every table was verified after import, and implementation findings were used to improve the database design where necessary.

# Import Strategy

The datasets were imported in an order that respected foreign key dependencies.

Entity and lookup tables were imported before transactional tables to ensure that referenced records already existed when foreign key constraints were enforced.

Import order:

1. customers
2. products
3. sellers
4. geolocation
5. product_category_name_translation
6. orders
7. order_items
8. order_payments
9. order_reviews

This approach eliminated foreign key violations during import.

---

# Validation Process

After importing each table, a validation query was executed to confirm that the expected number of records had been loaded successfully.

Example:

```sql
SELECT COUNT(*)
FROM customers;
```

This validation process was repeated for every table before continuing to the next import.

# Implementation Challenges

## Challenge 1 – geolocation

The `geolocation` table uses a surrogate key (`geolocation_id`) generated by PostgreSQL.

Because this column does not exist in the CSV file, it was excluded during import, allowing PostgreSQL to generate the identity values automatically.

### Resolution

Excluded the identity column during import while mapping the remaining CSV columns to the corresponding database columns.

---

## Challenge 2 – order_reviews

During import, PostgreSQL returned the following error:

```
duplicate key value violates unique constraint "pk_order_reviews"
```

Investigation revealed that duplicate `review_id` values exist in the source dataset and reference different orders.

The original assumption that `review_id` was unique proved incorrect.

### Resolution

The table was redesigned to use a surrogate key (`review_key`) as the primary key while preserving `review_id` as a business identifier.

This change allowed every source record to be imported without compromising entity integrity.

---

# Lessons Learned

This implementation reinforced several practical database concepts:

- Foreign key dependencies influence import order.
- Validation should be performed after every import.
- Database design assumptions must be tested against real data.
- Surrogate keys provide flexibility when natural keys cannot guarantee uniqueness.
- Data quality issues should be investigated before modifying the schema.

---

# Conclusion

The PostgreSQL implementation successfully imported and validated all nine tables while preserving compatibility with the original dataset.

The resulting database provides a reliable foundation for exploratory SQL, business analysis, and reporting.
