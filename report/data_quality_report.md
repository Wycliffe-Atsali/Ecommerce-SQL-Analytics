# Data Quality Report

## Overview

This report summarizes the data quality assessment performed after importing the Brazilian E-Commerce Public Dataset by Olist into PostgreSQL.

The objective was to verify that all datasets were imported successfully, validate referential integrity, identify potential data quality issues, and determine whether the database was suitable for exploratory analysis and business reporting.

# Primary Key Validation

Primary keys and composite primary keys were validated after import.

Validation confirmed that:

- Natural primary keys contain no duplicate values.
- Composite primary keys uniquely identify each record.
- Surrogate keys generated by PostgreSQL function correctly.

No unexpected duplicate primary keys were identified.

# Foreign Key Validation

Foreign key relationships were validated after data import.

The following relationships were successfully verified:

- orders → customers
- order_items → orders
- order_items → products
- order_items → sellers
- order_payments → orders
- order_reviews → orders

No orphan records were identified.

Referential integrity was successfully maintained throughout the database.

# NULL Value Assessment

Critical identifier columns were reviewed for unexpected NULL values.

Primary key columns contained no NULL values.

Several non-key columns intentionally contain NULL values.

Examples include:

- Product dimensions for products with incomplete catalog information.
- Review titles and review comments where customers submitted only numeric ratings.
- Delivery timestamps for canceled or undelivered orders.

These NULL values represent valid business scenarios rather than data quality problems.

---

# Schema Validation Findings

## order_reviews

During implementation, importing the `order_reviews` table revealed duplicate `review_id` values.

Further investigation confirmed that duplicate review identifiers reference different orders within the source dataset.

Because `review_id` is not guaranteed to be unique, it could not reliably serve as the table's primary key.

### Resolution

The schema was revised by introducing a surrogate key (`review_key`) as the primary key.

The original `review_id` was retained as a business identifier to preserve compatibility with the source dataset.

This change improved entity integrity without modifying the original data.

---

## geolocation

The original dataset does not contain a reliable natural primary key for geolocation records.

A surrogate key (`geolocation_id`) was introduced during database design.

During import, the identity column was excluded from CSV mapping so PostgreSQL could automatically generate unique values.
